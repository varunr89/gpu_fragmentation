\begin{abstract}
We integrate Fragmentation Gradient Descent (FGD) into Gavel's heterogeneity-aware scheduling framework through a two-stage design: Gavel's policy layer determines GPU type and quantity allocations via linear programming, while FGD selects the specific nodes that minimize fragmentation growth.
We replicate both systems independently, validating Gavel on the Microsoft Philly trace (504 experiments) and FGD on the Alibaba GPU-sharing trace (6 placement policies across 1,200 nodes with 5,592 GPUs), then evaluate the combined scheduler on two Alibaba cluster topologies under both Max-Min Fairness and FIFO policies (900 total experiments).
Our results reveal that fragmentation-aware placement reduces the fragmentation rate by 30--40\% on clusters with mixed node sizes (1--8 GPUs per node), increasing cluster utilization by up to 8 percentage points and cluster throughput by 17\% under high load, but provides no measurable benefit on uniform-node clusters, where all placement strategies converge.
This finding demonstrates that the value of fragmentation-aware scheduling depends on \textit{node topology diversity}, not GPU type heterogeneity, and that effective cluster scheduling requires jointly considering allocation policy, placement strategy, and physical topology.
\end{abstract}

\section{Introduction}
GPU cluster scheduling for deep learning has evolved rapidly as organizations deploy increasingly diverse hardware for training and inference workloads.
In production environments, GPU utilization averages between 25\% and 50\% \cite{FGD}, motivating the adoption of GPU sharing where multiple tasks run on a single physical GPU through partial resource allocation.

While GPU sharing improves aggregate utilization, it introduces \textit{resource fragmentation}: remaining capacity on individual devices may be too small to satisfy subsequent requests, and these fragments cannot be consolidated across physical GPU boundaries.
Production trace analysis from Alibaba clusters shows that fragmentation renders 21\% to 42\% of unallocated GPU resources unusable during normal operation \cite{ALIBABA, FGD}.

Separately, heterogeneous GPU clusters require scheduling policies that account for varying per-model speedups across accelerator types.
A ResNet-50 model sees a 10x throughput improvement moving from a K80 to a V100, while an A3C reinforcement learning model sees only 2x\cite{GAVEL}.

Two recent systems address complementary aspects of this problem.
Gavel \cite{GAVEL} expresses scheduling policies as optimization problems over an effective throughput abstraction, enabling heterogeneity-aware allocation with support for fairness, makespan minimization, and hierarchical policies.
However, Gavel was validated exclusively on whole-GPU workloads where fragmentation does not arise.
FGD \cite{FGD} defines a statistical fragmentation measure and places tasks toward the steepest descent of this measure, reducing unallocated GPUs by up to 49\%.
However, FGD operates under FIFO ordering without a flexible policy framework.

At the same time, these two mechanisms operate on different levels of abstraction. 
Gavel's LP reasons about aggregate capacity at the GPU-type level, while FGD 
makes node-level placement decisions based on the instantaneous distribution 
of free slots. This separation simplifies system design but also raises a natural question: whether limited coordination between allocation and placement could further improve packing efficiency when clusters become fragmented.

We show that the value of fragmentation-aware placement depends not on GPU type heterogeneity but on \textit{node topology diversity}, specifically the variance in GPUs per node across a cluster.
On clusters with mixed node sizes (1, 2, 4, and 8 GPUs per node), FGD-style placement reduces fragmentation by 30--40\% and recovers up to 280 GPUs of usable capacity.
On uniform-node clusters, all placement strategies converge and fragmentation-aware scheduling provides no measurable benefit.
This dimension is not evaluated by either Gavel or FGD in their original work.

\section{Background \& Related Work}

\subsection{Heterogeneity-Aware Scheduling with Gavel}
Modern GPU clusters contain multiple accelerator generations that exhibit widely varying per-model throughputs.
Gavel \cite{GAVEL} addresses this by expressing scheduling policies as optimization problems over allocation matrices, where each entry specifies the fraction of wall-clock time a job spends on a given accelerator type.

As illustrated in Fig.~\ref{fig:gavel_algorithm}, Gavel profiles each job's throughput on every GPU type, then solves a Max-Min Fairness LP to maximize the minimum effective throughput across all jobs, subject to per-type capacity constraints.
The resulting allocation assigns each job to the GPU types where it achieves the highest throughput while maintaining fair sharing across the cluster.
A round-based mechanism maps these fractional allocations to physical resource assignments using priority scores.

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{fig2_gavel_algorithm.png}
  \caption{Gavel's heterogeneity-aware allocation. Stage 1: Each job exhibits different throughputs across GPU types (e.g., ResNet runs fastest on V100, Transformer on P100). Stage 2: The Max-Min Fairness LP allocates time fractions per GPU type to maximize the minimum effective throughput. Stage 3: Each job receives GPUs on its best types -- ResNet-18 gets 1.3--2.1x speedup over a heterogeneity-agnostic baseline.}
  \label{fig:gavel_algorithm}
\end{figure}

Gavel generalizes policies including Least Attained Service (max-min fairness), finish-time fairness, FIFO, and makespan minimization.
However, Gavel was validated exclusively on the Microsoft Philly trace, which contains only whole-GPU job requests, and its placement logic does not consider fragmentation from partial GPU allocations.

\subsection{Fragmentation-Aware Placement with FGD}
GPU sharing introduces \textit{resource fragmentation}: remaining capacity on individual devices may be too small to satisfy subsequent requests, and these fragments cannot be recombined across physical device boundaries, unlike memory or disk fragmentation.
In Alibaba's production deployment, tasks receive partial GPU allocations at a granularity of 0.01 GPUs \cite{FGD}.

FGD \cite{FGD} addresses this through a statistical fragmentation measure that computes, for each node, the expected number of GPUs that cannot serve a randomly sampled task from the workload.
This captures both \textit{deficient} fragmentation (insufficient GPU capacity) and \textit{stranded} fragmentation (sufficient GPU but insufficient co-located CPU or memory).
As illustrated in Fig.~\ref{fig:fgd_algorithm}, FGD evaluates every eligible node for each incoming task, computes the fragmentation increment for each candidate placement, and selects the node that yields the minimum gradient.

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{fig3_fgd_algorithm.png}
  \caption{FGD placement algorithm. For a 2-GPU job, FGD computes the fragmentation gradient at each server. Server 4 (gradient = 0.08) is selected because placing the job there fully packs the server, creating zero new fragmentation. Random placement on Server 1 would scatter fragments across two servers.}
  \label{fig:fgd_algorithm}
\end{figure}

\subsection{Other Related Work}
GPU cluster scheduling has evolved along several complementary dimensions (see Fig.~\ref{fig:literature_genealogy} in the Appendix).
Tiresias~\cite{TIRESIAS} and Themis~\cite{THEMIS} address fairness and priority scheduling.
Gandiva~\cite{GANDIVA} introduces time-slicing and migration; HiveD~\cite{HIVED} provides topology-aware virtual clusters; Salus~\cite{SALUS} enables fine-grained GPU sharing.
Pollux~\cite{POLLUX} co-adapts resource allocation with training hyperparameters, and Sia~\cite{SIA} extends heterogeneity-aware scheduling with elastic scaling.
Our work unifies these two lines by combining policy-level allocation with placement-level fragmentation minimization, enabling evaluation of their interaction on production traces.


\section{Methodology}

Our approach proceeds in three phases: baseline replication, integration, and comparative evaluation.
A key challenge is that Gavel uses round-based simulation with dynamic arrivals and departures, measuring job completion time (JCT), while FGD uses workload inflation with monotonic fill, measuring fragmentation.
Our combined evaluation bridges both methodologies (see Fig.~\ref{fig:eval_methodology} in the Appendix).

\textbf{Phase 1: Baseline Replication.}
We replicate Gavel on the Microsoft Philly trace (108 GPUs: 36 V100, 36 P100, 36 K80) and replicate FGD on the Alibaba v2023 trace~\cite{ALIBABA} (Cluster H: 5,592 GPUs, 1,200 nodes with mixed node sizes of 1, 2, 4, and 8 GPUs).

\textbf{Phase 2: Integration.}
We integrate FGD into Gavel as a two-stage architecture (Fig.~\ref{fig:architecture}).
Gavel's policy layer solves an LP for time-fraction allocations per GPU type, and FGD then selects the physical node with the minimum fragmentation gradient within the allocated type.
This preserves policy-level optimization while adding fragmentation awareness.

The two stages, however, rely on different views of cluster state. The LP reasons about aggregate GPU-type capacity, while placement depends on node-level free-slot structure. We therefore also examine a lightweight coordination mechanism, presented later as an exploratory extension to the base Gavel+FGD design.

\textbf{Phase 3: Evaluation.}
We test on two Alibaba cluster topologies to isolate the effect of node structure:

\section{Replication Results}

\subsection{Gavel Replication}
We ran 504 experiments across three configurations from the original Gavel paper~\cite{GAVEL}: single-GPU jobs (MMF), multi-GPU jobs (MMF), and multi-GPU jobs (Finish-Time Fairness), each sweeping arrival rate across 3 seeds.

Fig.~\ref{fig:gavel_replication} overlays our results against digitized paper references.
Gavel's heterogeneity-aware policy reduces average JCT by 20--70\% compared to the baseline.
Sub-saturation error is small (3.05 hours RMSE); larger deviations appear only near the saturation knee where our simulator saturates at a slightly different load point than the paper reference, likely due to underspecified queue management details.

\subsection{FGD Replication}
We replicate FGD's standalone evaluation on Alibaba Cluster H using the inflation methodology~\cite{FGD}, comparing six placement policies.
Fig.~\ref{fig:fgd_replication} shows our replication of four key metrics alongside digitized references.
FGD maintains the lowest fragmentation throughout the demand range.
At 80\% demand, Random placement fragments 85\% of unallocated GPUs while FGD fragments only 42\%.
The fragmentation-to-total ratio stays below 8\% with FGD, and relative error subplots confirm close agreement across the full demand range.

To test robustness beyond the inflation setting, we also ran a standalone task-arrival-order study on Alibaba at 90\% cluster scale using seven arrival orders (trace, ascending, descending, and four phased variants). We compared Random, Clustering, and FGD under identical total demand. Across these runs, FGD showed strong order sensitivity: it degraded relative to Clustering in most orders, but became best in one phased-order configuration (Table~\ref{tab:arrival_order_pending}). This indicates that replication outcomes depend not only on scheduler identity but also on task arrival order assumptions.


\section{Combined Gavel+FGD Results}

We evaluate the integrated system (Fig.~\ref{fig:architecture}) on both Alibaba cluster topologies under four placement strategies: Random, Strided (Gavel's default sequential assignment), BestFit, and FGD.

\subsection{Fragmentation Across Arrival Rates}
Fig.~\ref{fig:frag_comparison} compares fragmentation rate versus job arrival rate across both cluster topologies under Max-Min Fairness with four placement strategies.
On Cluster H (panel a), Strided placement produces the highest fragmentation (9--10\% at 300+ jobs/hr), while BestFit and FGD reduce it to 4--6\% -- a consistent 40--50\% reduction across the full arrival rate sweep.
Random placement falls between these two groups.
This ordering holds consistently: FGD $\approx$ BestFit $<$ Random $<$ Strided, under both Max-Min Fairness and FIFO, indicating that the placement benefit is \textit{independent of the allocation policy}.

On the Alibaba Split cluster (panel b), the gap narrows substantially: all strategies converge to within 1--2 percentage points.
The residual separation arises because even uniform 8-GPU nodes exhibit fragmentation when partial-GPU jobs leave mismatched remainders, but this effect is far smaller than the mixed-node case.

  \label{fig:frag_comparison}

\subsection{Utilization Impact of Placement}
Fragmentation is not just a cosmetic metric, but it also directly reduces cluster capacity.
Fig.~\ref{fig:util_impact} shows that on Cluster~H, Strided placement caps at 89\% utilization under high load while BestFit and FGD reach 94\%, a gap of 5 percentage points, representing approximately 280 GPUs of recoverable capacity on a 5,592-GPU cluster.
At typical cloud GPU rates of \$2--3 per GPU-hour on average \cite{GCPGPU}, recovering 280 GPUs translates to approximately \$400K--\$600K per month in average reclaimed capacity, illustrating that placement-level fragmentation carries substantial cost at production scale.
Panel~(b) shows \textit{effective capacity} (utilization minus fragmentation rate), which captures the fraction of the cluster doing useful work.
At 250 jobs/hr, Strided achieves 64\% effective capacity while BestFit reaches 78\% -- a 13-point gap, translating to 17\% more completed jobs.

Individual job completion times, measured over the steady-state window (jobs 4000 to 5000), are identical across all placements to within numerical precision.
Placement affects cluster-level capacity but not per-job throughput, confirming that allocation and placement operate on orthogonal dimensions.

\section{Analysis}

\subsection{The Role of Node Topology}
The contrast between panels (a) and (b) in Fig.~\ref{fig:frag_comparison} reveals the central finding of our evaluation: \textbf{fragmentation-aware placement matters only when clusters have diverse node sizes.}

Fig.~\ref{fig:topology} illustrates why.
On uniform-node clusters (left panel), any job that fits on one node fits equally well on any other node of the same type.
The placement decision is trivial because all choices produce the same fragmentation.
On mixed-node clusters (right), a 3-GPU job cannot use scattered 1-GPU fragments across different nodes, so placement decisions directly determine whether capacity remains usable.

This finding has practical implications.
Cloud providers deploying new clusters with uniform node sizes (e.g., standard 8-GPU servers) gain little from fragmentation-aware placement.
However, clusters that accumulate heterogeneous hardware over time, mixing older 1--2 GPU machines with newer 4--8 GPU servers, benefit substantially from FGD-style placement.

\subsection{Policy Independence of Placement Effects}
Our evaluation under both Max-Min Fairness and FIFO allocation policies shows that FGD's placement benefit is consistent regardless of the allocation policy.
On Cluster H, the fragmentation reduction from FGD is 30--40\% under both policies, with nearly identical curves (see Appendix Fig.~\ref{fig:combined_cluster_h} and Fig.~\ref{fig:combined_uniform} for per-utilization breakdowns).
Combined with the utilization impact shown in Fig.~\ref{fig:util_impact}, this confirms that allocation and placement are truly orthogonal: placement determines cluster capacity, while allocation determines per-job throughput.

\subsection{Gavel at Alibaba Scale}
Running Gavel's LP-based allocation on the Alibaba cluster (6,200 GPUs, up to 1,700 concurrent jobs) revealed performance bottlenecks not present at Philly scale (108 GPUs, 50 jobs).
At Alibaba scale, the LP solver consumes 48\% of per-round runtime (3.0s per solve), compared to 8\% at Philly scale.
We addressed this through solver warm-starting and logging optimizations, reducing per-round time from 1.58s to 0.64s -- sufficient for our evaluation but highlighting that Gavel's LP formulation faces scalability limits at production cluster sizes.

\subsection{Improving LP Allocation with Placement Opportunity Signals}
% Jiayu: distribution shift experiments and additional policy analysis

While allocation and placement largely operate on orthogonal dimensions, the two layers do not observe the same state: the LP reasons about aggregate GPU-type capacity, whereas placement operates on node-level free-slot structure. 
To explore whether limited coordination between these layers can improve packing efficiency, we evaluate a variant that augments the LP objective with a \textit{Placement-Opportunity-Aware} (POA) signal derived from the current cluster state.

Additional results comparing Gavel, GavelFGD, and GavelFGD+ across utilization levels are shown in Appendix Fig.~\ref{fig:gavelfgd_absolute}.

\textbf{Fragmentation reduction.}
FGD placement alone already substantially reduces fragmentation. Compared to the Gavel baseline, GavelFGD lowers fragmentation by roughly 40--50\% across most utilization levels.
GavelFGD+ further improves these results at high load. When cluster utilization exceeds $\sim$75\%, fragmentation remains consistently lower than GavelFGD.

At 78.8\% utilization (310 jobs/hour), fragmentation decreases from 1.04\% to 0.67\% (+35.5\%).
At 85.4\% utilization (360 jobs/hour), fragmentation drops from 2.01\% to 0.47\%, a 76.7\% reduction relative to GavelFGD and an 86.6\% reduction relative to the original Gavel scheduler.

\textbf{Cluster utilization.}
Fig.~\ref{fig:gavelfgd_absolute}(c) shows that the percentage of unallocated GPUs remains nearly identical across all three systems.
This confirms that the observed improvements do not come from additional resources, but rather from more efficient packing of jobs across nodes.

\textbf{POA-guided allocation.}
GavelFGD+ introduces a \textit{Placement-Opportunity-Aware} (POA) signal into the LP objective.
For each GPU type $t$ and job demand $d$, the scheduler computes:

\[
\text{poa\_score}[t,d] =
\frac{\#\text{nodes with } \ge d \text{ free GPUs}}
{\#\text{nodes of type } t}.
\]

This score represents the fraction of nodes that can immediately accommodate a job of size $d$. 
The LP objective becomes:

\[
\max \min_j \text{normalized\_throughput}(j)
+
\mu \sum_{j,t} \text{poa\_score}[t,d_j] \cdot x_{j,t},
\]

where $x_{j,t}$ is the fraction of time job $j$ is allocated to GPU type $t$.

Intuitively, this term encourages the LP to steer flexible jobs toward GPU types where placement is currently more feasible.
When utilization is low, most nodes contain large free slots and POA scores remain similar across GPU types, leaving the LP solution largely unchanged.
As utilization increases and large contiguous slots become scarce, POA scores diverge, allowing the LP to guide allocations toward GPU types where placement is more likely to succeed.

This behavior is particularly relevant for the Alibaba workload used in our experiments.
G2 GPUs account for roughly 71\% of cluster capacity, and most jobs can run only on G2.
As a result, only multi-type-compatible jobs influence the LP allocation.
The POA signal steers these flexible jobs toward GPU types with higher placement opportunity, helping preserve larger contiguous GPU blocks and reducing fragmentation under heavy load.

\section{Limitations}
% talk simultaion, policy limitations, allow space for previous FGD sensitivity
Our evaluation relies entirely on trace-driven simulation, which introduces several sources of potential bias.

\textbf{Simulation Fidelity.}
As CausalSim~\cite{CAUSALSIM} demonstrates, trace-driven simulation assumes that the intervention being evaluated does not affect the validity of the replayed trace.
This \textit{exogenous trace assumption} may not hold in practice: scheduling decisions can alter queuing delays and resource contention in ways that feed back into job performance.
For example, a placement decision that increases co-located interference could change a job's effective throughput, invalidating the profile recorded under a different policy.
Our simulator does not model such feedback effects, and the magnitude of this bias in GPU scheduling traces remains an open question.

% ======== Kim: Review point below (should summarize your findings) =====
\textbf{Workload Stationarity.}
FGD's fragmentation measure assumes a fixed task-popularity distribution, and our arrival-order study (Table~\ref{tab:arrival_order_pending}) shows that its effectiveness degrades when the workload mix shifts.
This helps explain why Monte Carlo evaluation can make FGD look consistently strong: sampling with replacement smooths arrivals toward a stationary distribution.
In production, however, workload composition evolves over time, so fixed-prior evaluation may overstate FGD's benefit under distributional drift.
In Gavel+FGD, round-based allocation partly mitigates this effect by adapting job eligibility each round, but node placement still relies on a fixed workload prior.


\textbf{Migration and Preemption Costs.}
Gavel's round-based mechanism assumes jobs can be checkpointed and migrated between GPUs with negligible overhead.
In practice, migration incurs costs from checkpoint serialization, data transfer, and GPU warm-up that vary by model size and framework.
Our simulator does not account for these costs, which could reduce the effective benefit of both heterogeneity-aware allocation and fragmentation-aware placement in production settings.



\section{Conclusion}

We presented an integrated GPU cluster scheduler combining Gavel's heterogeneity-aware allocation with FGD's fragmentation-aware placement.
Through replication of both systems (504 Gavel experiments on Philly, standalone FGD on Alibaba) and 900 combined experiments across two cluster topologies, we demonstrated that:

\begin{enumerate}
\item Gavel's heterogeneity-aware policies reduce JCT by 20--70\% compared to heterogeneity-agnostic baselines, and this benefit holds at Alibaba scale.
\item FGD placement reduces fragmentation by 30--40\% on clusters with mixed node sizes (1--8 GPUs per node), translating to up to 8 percentage points higher cluster utilization and 17\% more completed jobs under high load.
\item Fragmentation-aware placement provides no benefit on uniform-node clusters, where all strategies converge.
\end{enumerate}

The central insight is that fragmentation in GPU clusters is driven by \textit{node topology diversity}, specifically the variance in GPUs per node, rather than GPU type heterogeneity.
This suggests that cluster architects can mitigate fragmentation either through hardware homogeneity (uniform node sizes) or through software (FGD-style placement).
Our two-stage design demonstrates that allocation policy and placement strategy address orthogonal concerns and can be composed independently, while limited coordination between the two layers may further improve packing efficiency under heavy fragmentation.

Future work includes extending the evaluation to GPU-sharing workloads with partial allocations (sub-GPU granularity), evaluating migration-aware policies that account for job relocation costs, profiling the combined system under production-scale job churn, keeping Gavel's allocation loop while making placement priors adaptive, and exploring richer forms of coordination between allocation and placement beyond the lightweight placement-opportunity signal studied here.

\section{Wall Time Analysis}
Fig.~\ref{fig:wall_time} reports simulation wall time across all experiment sets, totaling approximately 850 experiments on Stanford FarmShare (Intel Xeon, no GPU required for simulation).

Panel (a) shows Gavel replication experiments on the Philly trace (108 GPUs).
Wall time increases exponentially near the saturation knee (2--3 jobs/hr for multi-GPU, 6--7 for single-GPU), where experiments hit the 24-hour SLURM time limit.
Sub-saturation runs complete in under 1 hour.

Panel (b) shows the combined Gavel+FGD experiments on Cluster H (5,592 GPUs).
Wall time remains flat at 1--6 minutes across all arrival rates, reflecting the relatively fast LP solves at this scale.
All placement strategies have comparable wall time.

Panel (c) shows Alibaba Split experiments (6,212 GPUs, 12 heterogeneous types).
Wall time decreases from 5+ minutes at low load to under 1 minute at high load.
The higher cost at low arrival rates reflects the LP solver processing more scheduling rounds before the measurement window completes.

\section{Visualization Tool}
To support interactive exploration of scheduling behavior across 843 experiments, we developed a web-based visualization tool (Fig.~\ref{fig:viz_tool}).
The tool loads experiment telemetry from a custom binary format hosted on Azure Blob Storage and decodes it in a Web Worker for responsive interaction.

Key features include:
\begin{itemize}
\item \textbf{Side-by-side comparison} of two experiments with synchronized playback (Fig.~\ref{fig:viz_tool}, top), enabling direct visual comparison of placement strategies under identical workloads.

The tool was instrumental in debugging saturation detection, validating replication accuracy against paper figures, and identifying the node-topology dependence of fragmentation.

\section{Task Arrival Order Study}
Table~\ref{tab:arrival_order_pending} reports total pending task GPUs when cumulative arrived workload reaches 96\% of cluster GPU capacity. Higher values indicate more unscheduled GPU demand at the same near-saturation load point.

We evaluate four task-arrival-order families:
\begin{itemize}
\item \textbf{Trace}: original creation-time order from the Alibaba trace.
\item \textbf{Ascending}: tasks sorted from smaller to larger GPU requests.
\item \textbf{Descending}: tasks sorted from larger to smaller GPU requests.
\item \textbf{Phased}: tasks grouped by GPU-demand tier and submitted phase by phase.
\end{itemize}

The phased tiers are:
\begin{itemize}
\item Tier 0: $gpu = 0$ (CPU-only tasks)
\item Tier 1: $0 < gpu < 0.5$ (small fractional GPU)
\item Tier 2: $0.5 \le gpu < 1$ (large fractional GPU)
\item Tier 3: $gpu = 1$ (single full GPU)
\item Tier 4: $gpu > 1$ (multi-GPU tasks)
\end{itemize}
